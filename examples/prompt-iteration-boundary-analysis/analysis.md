# 案例分析：提示词迭代中的边界情况处理

## 案例概述

**场景**：智能数据推荐助手工作流

> 实际案例来源：数据对比分析工作流 LLM 节点

**症状**：外部实体无指标数据时，与基准对比的数值显示负数

**预期输出**：`-`（表示无数据）  
**实际输出**：`-131.50元/单位 | 差异率：-100.00%`

**根因分析**：
```
提示词规则：
"空值按0计算，差异率 = (外部 - 基准) / 基准 × 100%"

外部无数据时：
- 外部量化值 = 0（空值转0）
- 差值 = 0 - 131.50 = -131.50
- 差异率 = -100%

问题：将"无数据"(L1)当作"空数据"(L2)处理
```

### 问题2：无意义前缀
**症状**：数据整合输出中出现"示例数值："（无值）

**预期输出**：``（空字符串）  
**实际输出**：`示例数值：`

**根因分析**：
```
提示词规则：
"为所有指标值补充「项目名称+指标名：」前缀"

空值处理：
- 输入：`{"数值": ""}`
- 处理：直接拼接前缀 → `"示例数值：" + ""`
- 输出：`"示例数值："`

问题：未判断值是否为空就添加前缀
```

## 诊断过程

### Phase 1: 日志分析

**执行概览**：
- 总耗时：568秒
- Token消耗：58,888
- 循环次数：8次

**关键节点耗时**：
| 节点 | 耗时 | 说明 |
|-----|------|------|
| 循环5-结果封装 | 231秒 | 指标搜索+封装逻辑复杂 |
| 循环6-对比分析 | 64秒 | **问题1根因** |
| 循环7-数据整和 | 82秒 | **问题2根因** |

### Phase 2: 提示词分析

**对比分析节点关键规则**：
```markdown
2. **空值标注规则**：
   ✅ 当指标「原始值为空字符串」时，量化值输出"-"
   ✅ 仅一方为空时，空值方按0计算

缺陷：未区分"无数据"(对象不存在)和"空数据"(对象存在但值为空)
```

**数据整和节点关键规则**：
```markdown
1. **实体分组重构**：
- 内部实体：按实体名称全局分组，多项目合并
- 合并规则：在指标值前补充「项目名称+指标名：」前缀

缺陷：未判断指标值是否为空就添加前缀
```

### Phase 3: 迭代轨迹追踪

**v1: 初始版本**
```markdown
规则：为所有指标值补充「项目名称+指标名：」前缀
```
- 问题：空值也被添加前缀

**v2: 用户反馈"去掉前缀"**
```markdown
规则：不添加任何前缀
```
- 修复了问题2
- 新问题：内部实体多项目场景无法区分数据来源

**v3: 用户反馈"内部要区分"**
```markdown
规则：
- 外部实体：不添加前缀
- 内部实体：
- 单项目：不添加前缀
- 多项目有差异：添加「项目名称：」前缀
- 多项目无差异：不添加前缀
```
- 引入条件化逻辑
- 需要增加差异检测机制

## 诊断方法应用

### 数据状态三层分类法

| 层级 | 场景 | 处理策略 |
|------|------|----------|
| L1: 无数据 | 外部实体无任何指标 | 输出"-"，不参与计算 |
| L2: 空数据 | 指标对象存在但全为空 | 按业务规则处理 |
| L3: 部分数据 | 部分指标有值、部分为空 | 空值按0计算 |

**问题1修复**：将外部无数据识别为L1，而非L2/L3

### 条件化输出设计模式

```markdown
❌ 反模式：
"为所有字段添加前缀"

✅ 正模式：
"判断值是否非空：
 - 是：添加前缀
 - 否：保留空值"
```

**问题2修复**：添加空值判断后再拼接前缀

### 提示词逻辑验证四问

**Q1: 边界覆盖**
- 问：是否处理了空值、无数据、部分数据场景？
- 答：v1未覆盖，v2/v3逐步完善

**Q2: 条件完备**
- 问：if/else是否覆盖所有分支？
- 答：v3增加了多项目差异判断，覆盖更全

**Q3: 一致性**
- 问：规则描述与示例输出是否一致？
- 答：v1不一致（规则说"所有"，但示例有空值）；v3已对齐

**Q4: 可验证**
- 问：是否提供了测试用例？
- 答：建议在提示词中增加输入/输出示例对

## 关键洞察

### 1. "补丁叠补丁"的风险
每次用户反馈后只修局部，未考虑整体一致性，导致：
- v1→v2：修复空值问题，但破坏了多项目场景
- v2→v3：修复多项目问题，需要增加复杂判断

### 2. 兜底规则的重要性
原始提示词有兜底规则：
```markdown
"所有规则执行均不得修改、伪造用户输入的任何原始字段值"
```
但"添加前缀"规则与之冲突，说明兜底规则被覆盖。

### 3. 演进路径的通用模式
```
全加 → 全不加 → 条件化
```
这是提示词优化的典型轨迹，建议在设计之初就考虑条件化。

## 可复用检查清单

### 数据类节点诊断
```markdown
□ 数据源对象是否存在？（null/undefined检查）
□ 字段值是否为空？（""/null/undefined区分）
□ 多数据源是否合并？（差异检测机制）
□ 字符串拼接前是否判空？
□ 默认值是否合理？（避免0/-1歧义）
```

### 提示词迭代管理
```markdown
□ 当前修改是否影响已有场景？
□ 兜底规则是否被覆盖？
□ 新增条件是否覆盖所有分支？
□ 示例输出是否更新？
□ 是否提供测试用例？
```
