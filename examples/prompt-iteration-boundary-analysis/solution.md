# 解决方案：提示词迭代中的边界情况处理

## 修复后的提示词

### 节点1：对比分析

```markdown
### 数据对比分析提示词

请按照以下要求完成数据对比分析：

#### 一、核心量化规则

1. **数据状态三层分类（关键修复）**：
   
   | 层级 | 定义 | 处理策略 |
   |------|------|----------|
   | **L1: 无数据** | 指标对象不存在/null/无字段 | 量化值输出"-"，不参与对比计算 |
   | **L2: 空数据** | 指标对象存在但所有值为空字符串 | 按业务规则处理（可能参与计算） |
   | **L3: 部分数据** | 部分指标有值、部分为空 | 空值按0参与计算 |
   
2. **空值标注规则**：
   ✅ L1（无数据）：「外部量化值」输出"-"，「量化对比结果」输出"-"
   ✅ L2/L3（有空值）：空值方量化值输出"-"，按0参与对比计算
   ✅ 双方均有值：正常计算差值和差异率

3. **计算规则**：
- 差值 = 外部量化值 - 基准量化值
- 差异率 = (差值 / 基准量化值) × 100%（保留2位小数）
   - **严禁将L1（无数据）按0计算**

#### 二、输出示例

```json
{
"数值": {
"基准量化值": "131.50元/单位",
    "外部量化值": "-",
    "量化对比结果": "-"
  }
}
```
```

### 节点2：数据整和

```markdown
### 数据信息整合提示词

#### 一、核心处理规则

0. **最高优先级兜底规则**
   - 所有规则执行均不得修改、伪造用户输入的任何原始字段值
   - 仅允许新增「推荐等级」「推荐理由」2个字段

0.5 **条件化前缀规则（关键修复）**
   
**内部实体**：
- 判断该实体是否涉及**多个不同项目**
- **单项目**：指标值保持原样，**不添加任何前缀**
- **多项目**：判断指标值是否存在差异
- ✅ **有差异**：添加「项目名称：」前缀，多项目用「，」分隔
     - ✅ **无差异**：不添加前缀，直接输出指标值（去重）
   
**外部实体**：
   - **不添加任何前缀**
   - 严格保留「项目为外层独立对象」的数组结构

1. **字符串拼接前置检查**：
   - 拼接前判断值是否为空字符串/null
   - **空值不拼接任何前缀**

#### 二、输出示例

**内部实体-单项目**：
```json
{
  "指标值": {
    "数值": "100.00",
    "交期": "5天"
  }
}
```

**内部实体-多项目有差异**：
```json
{
  "指标值": {
    "数值": "项目A：95.00，项目B：110.50",
    "指标值": "X%"
  }
}
```
（注：指标值无差异，不加前缀）

**内部实体-多项目无差异**：
```json
{
  "指标值": {
    "数值": "项目A：100.00，项目B：110.00",
    "指标值": "X%"
  }
}
```
（注：指标值相同，不加前缀）

**外部实体**：
```json
{
  "外部实体": [
    {
      "项目名称": "项目A",
      "指标值": {
        "数值": "88.50",
        "交期": "7天"
      }
    }
  ]
}
```
（注：不加任何前缀）
```

## 修复要点

### 1. 引入数据状态三层分类

**修复前**：
```markdown
"空值按0计算"
→ 无数据和空数据都按0处理，导致负数
```

**修复后**：
```markdown
"L1无数据：输出'-'不参与计算
 L2空数据：按业务规则处理
 L3部分数据：空值按0计算"
→ 区分处理，避免负数
```

### 2. 条件化前缀注入

**修复前**：
```markdown
"为所有指标值补充前缀"
→ 空值也被添加前缀，输出"字段名："
```

**修复后**：
```markdown
"先判断：
- 单项目？不加
- 多项目有差异？加前缀
- 多项目无差异？不加
 - 值为空？不加"
→ 只在需要时添加前缀
```

### 3. 一致性验证

**检查点**：
- [x] 规则描述与示例输出一致
- [x] 覆盖空值、有值、差异、无差异四种场景
- [x] 提供明确的测试用例
- [x] 兜底规则未被覆盖

## 验证方法

### 测试用例1：外部实体无数据

**输入**：
```json
{
  "外部实体": {
    "指标值": null
  },
  "基准实体": {
    "数值": "131.50元/单位"
  }
}
```

**预期输出**：
```json
{
  "外部量化值": "-",
  "量化对比结果": "-"
}
```

### 测试用例2：空值指标

**输入**：
```json
{
  "指标值": {
    "数值": ""
  }
}
```

**预期输出**：
```json
{
  "指标值": {
    "数值": ""
  }
}
```
（无前缀）

### 测试用例3：多项目有差异

**输入**：
```json
{
  "项目": ["A", "B"],
  "指标值": {
    "A": {"价格": "10.0"},
    "B": {"价格": "11.3"}
  }
}
```

**预期输出**：
```json
{
  "价格": "A项目：10.0，B项目：11.3"
}
```

### 测试用例4：多项目无差异

**输入**：
```json
{
  "项目": ["A", "B"],
  "指标值": {
    "A": {"指标值": "X%"},
    "B": {"指标值": "X%"}
  }
}
```

**预期输出**：
```json
{
  "指标值": "X%"
}
```
（无前缀，去重）

## 最佳实践总结

### 1. 数据状态优先判定
在编写数据处理类提示词时，首先明确数据状态的层级：
```
无数据(L1) → 空数据(L2) → 部分数据(L3) → 完整数据(L4)
```

### 2. 条件化优于一刀切
避免"为所有..."的表述，改用"判断条件后..."：
```markdown
❌ "为所有字段添加前缀"
✅ "判断值非空后添加前缀"
```

### 3. 版本演进管理
建立提示词版本日志：
```markdown
## 版本历史
- v1.0: 初始版本，全加前缀
- v1.1: 修复空值问题，全不加前缀
- v2.0: 增加条件化逻辑，区分单/多项目
```

### 4. 测试驱动设计
在提示词中内置测试用例：
```markdown
### 测试场景
1. 空值输入 → 预期输出...
2. 有值输入 → 预期输出...
3. 差异输入 → 预期输出...
```
