# Dify 日志诊断方法论

> 系统化诊断 Dify 工作流执行问题的框架。

---

## 诊断原则

### 1. 诚实原则
- ✅ 有证据才下结论
- ⚠️ 证据不足时明确说明「需要验证」
- ❌ 绝不瞎编、绝不推测

### 2. 分层诊断
从外到内，逐层排查：
1. **配置层**：策略、参数设置
2. **绑定层**：工具是否正确绑定
3. **模型层**：模型兼容性
4. **提示词层**：指令明确性
5. **架构层**：Dify 版本限制

### 3. 止损机制
- 单次操作最多重试 3 次
- 总耗时超过 60 秒停止
- 连续 2 次失败立即停止

---

## 通用诊断流程

```
收到日志链接
    ↓
Phase 0: 前置检查
    - 检查 URL 有效性
    - 检查配置文件
    - 确认分析模式
    ↓
Phase 1: 日志采集
    - 导航到日志页面
    - 展开执行详情
    - 提取节点列表
    ↓
Phase 2: 节点分析
    - 通用节点检查
    - 专项诊断（如 Agent 问题）
    ↓
Phase 3: 报告生成
    - 选择报告模板
    - 输出诊断结论
```

## 节点类型诊断速查

| 节点类型 | 常见问题 | 检查要点 |
|---------|---------|---------|
| **LLM** | 输出不符合预期 | 提示词、模型参数、温度设置 |
| **Agent** | 工具不调用 | 策略、工具绑定、enable_thinking |
| **工具** | 执行失败 | 参数格式、认证信息、网络连接 |
| **代码执行** | 运行错误 | 代码逻辑、输入变量、超时设置 |
| **知识检索** | 检索不到 | 知识库配置、查询语句、相似度阈值 |
| **条件分支** | 判断错误 | 条件表达式、变量值、数据类型 |
| **循环** | 无限循环/不循环 | 循环条件、最大迭代次数、退出条件 |
| **HTTP 请求** | 请求失败 | URL、方法、Headers、Body、超时 |

## 报告模板选择

根据诊断完整度选择模板：

- **完整诊断**（90-100分）：所有检查项通过，问题明确
- **基本诊断**（70-89分）：主要问题明确，部分细节需验证
- **部分诊断**（50-69分）：问题方向明确，需补充信息
- **手动验证**（<50分）：信息不足，需用户手动检查

## 【新增】数据状态三层分类法

> 超越"空值vs有值"的二元思维，精细化处理数据边界情况

### 数据状态层级

| 层级 | 定义 | 处理策略 | 示例 | 常见错误 |
|------|------|----------|------|----------|
| **L1: 无数据** | 对象不存在/null/undefined | 不参与计算，输出"-" | `indicators: null` | 按0计算产生负数 |
| **L2: 空数据** | 对象存在但全为空值 | 按业务规则处理（可能参与计算） | `indicators: {price: "", qty: ""}` | 添加无意义前缀 |
| **L3: 部分数据** | 对象存在，部分有值部分空 | 空值按默认值/0计算 | `indicators: {price: "100", qty: ""}` | 忽略空值导致计算错误 |
| **L4: 完整数据** | 对象存在且所有字段有值 | 正常计算 | `indicators: {price: "100", qty: "10"}` | - |

### 诊断检查清单

```markdown
□ L1 检查：数据源对象是否存在？
  - 检查 null/undefined/缺失字段
  - 处理策略：输出"-"，不参与计算

□ L2 检查：对象存在但是否全为空？
  - 遍历所有字段判断
  - 处理策略：按业务规则，可能输出空或默认值

□ L3 检查：是否存在部分字段有值？
  - 识别哪些字段有值、哪些为空
  - 处理策略：空值按0或默认值参与计算

□ L4 检查：所有字段是否都有值？
  - 正常业务逻辑处理
```

### 典型故障场景

**场景1：负数输出**
- 症状：对比计算显示负数
- 根因：L1（无数据）被当作L2/L3（空数据/部分数据）处理，按0参与计算
- 修复：区分"无数据"和"数据为空"

**场景2：无意义前缀**
- 症状：输出"字段名："（无值）
- 根因：L2（空数据）被强制添加前缀
- 修复：添加前缀前判断值是否非空

### 可视化对比示例

**输入数据 vs 处理方式 vs 输出结果**

| 数据状态 | 输入示例 | 错误处理 | 错误输出 | 正确处理 | 正确输出 |
|---------|---------|---------|---------|---------|---------|
| **L1** | `external: null` | 按0计算 | `差值: -131.50` | 输出"-" | `外部量化值: -` |
| **L2** | `price: ""` | 直接加前缀 | `价格：` | 判空后处理 | `price: ""` |
| **L3** | `{p: "100", q: ""}` | 忽略空值 | `总价: 100` | 空值按0算 | `总价: 100×0=0` |
| **L4** | `{p: "100", q: "10"}` | 正常计算 | `总价: 1000` | 正常计算 | `总价: 1000` |

**关键判断逻辑**:
```javascript
if (data === null || data === undefined) {
  // L1: 无数据
  return "-";
} else if (isAllFieldsEmpty(data)) {
  // L2: 空数据
  return handleEmptyData(data);
} else if (hasPartialData(data)) {
  // L3: 部分数据
  return calculateWithDefaults(data);
} else {
  // L4: 完整数据
  return normalCalculate(data);
}
```

## 【新增】条件化输出设计模式

> 避免"一刀切"的字符串拼接和数据处理

### 反模式 vs 正模式

```markdown
❌ 反模式：
"为所有字段添加前缀：{{field}}"
→ 导致空值也被添加前缀，输出"字段名："

✅ 正模式：
"判断是否满足条件X：
 - 是：添加前缀{{prefix}}{{field}}
 - 否：保留原值{{field}}"
→ 条件可以是：字段值非空、多数据源存在差异、特定业务标识位
```

### 通用检查清单

```markdown
□ 字符串拼接前是否有空值判断？
  - 空字符串/null/undefined 都不应参与拼接

□ 多数据源合并时是否有差异检测？
  - 相同值：去重，不加标识
  - 不同值：分别标识
  - 部分为空：仅标识有值部分

□ 默认值设置是否合理？
  - 避免0/-1等引起歧义的默认值
  - 使用null或明确标识"无数据"

□ 输出示例是否覆盖所有情况？
  - 空值场景
  - 有值场景
  - 差异场景
  - 无差异场景
```

## 【新增】提示词版本演进追踪

> 管理多轮迭代中的逻辑一致性问题

### 典型演进轨迹

```
v1: 全加逻辑
    ↓ 用户反馈：空值不需要加
v2: 全不加逻辑
    ↓ 用户反馈：多项目场景需要区分
v3: 条件化逻辑 ← 当前最优解
    - 单项目：不加
    - 多项目有差异：加标识
    - 多项目无差异：不加
```

### 演进检查清单

```markdown
□ 兜底规则检查
  - 最早设定的"最高优先级规则"是否被后续规则覆盖？
  - 示例："不修改任何原始字段" vs "添加前缀"

□ 条件完备性检查
  - 新版本是否覆盖了所有分支？
  - if/else/elif 是否遗漏边界情况？

□ 一致性检查
  - 规则描述与输出示例是否一致？
  - 不同版本间是否存在冲突？

□ 可验证性检查
  - 是否提供了测试用例验证新逻辑？
```

## 【新增】S.V.E.R. 调试流程

> 标准调试流程：Step Run → Variable Inspection → Execution History → Root Cause Analysis

### S - Step Run（单步执行）

**目的**：定位问题节点

**操作步骤**：
1. 在 Dify 工作流编辑器中启用"单步执行"模式
2. 逐个节点运行，观察执行结果
3. 在问题节点暂停，记录现象

**关键检查点**：
- 该节点的输入是否符合预期？
- 该节点的输出是否异常？
- 执行时间和Token消耗是否正常？

### V - Variable Inspection（变量检查）

**目的**：对比输入/输出差异

**操作步骤**：
1. 打开"变量检查器"面板
2. 查看该节点的输入变量值
3. 查看该节点的输出变量值
4. 对比差异

**关键检查点**：
- 上游节点的输出是否正确传递？
- 数据类型是否一致？
- 是否存在未预期的空值？

**快速测试**：
- 直接在变量检查器中修改变量值
- 重新运行下游节点验证

### E - Execution History（执行历史对比）

**目的**：识别数据变化点

**操作步骤**：
1. 打开"执行历史"面板
2. 找到一次成功的执行记录
3. 与失败的执行进行对比
4. 找出关键差异

**关键检查点**：
- 相同的输入是否产生不同的输出？
- 时间戳：问题何时开始出现？
- Token消耗：是否有异常变化？

### R - Root Cause Analysis（根因分析）

**目的**：确定问题根源

**分析方法**：
1. **向上追溯**：问题节点的输入从哪来？
2. **分层诊断**：使用6层诊断法逐层排查
3. **假设验证**：提出假设，设计验证实验

**输出**：
- 问题根因定位
- 影响范围评估
- 修复方案建议

## 【新增】数据源差异对比框架

> 多数据源场景下的标准化对比方法

### 对比维度

#### 1. 结构差异
```markdown
□ 字段名是否一致？
  - 同义不同名：price vs unit_price
  - 字段缺失：外部数据缺少字段tax

□ 数据层级是否一致？
  - 扁平结构 vs 嵌套结构
  - 数组 vs 单对象
```

#### 2. 内容差异
```markdown
□ 相同字段值是否相同？
  - 单位差异：元 vs 万元
  - 格式差异：2024-01-01 vs 2024/01/01
  - 精度差异：10.0 vs 10.00

□ 空值表示方式是否一致？
  - "" vs null vs undefined vs "N/A"
```

#### 3. 处理差异
```markdown
□ 不同数据源是否有差异化的处理逻辑？
  - 内部实体：全局合并
  - 外部实体：按项目独立

□ 合并策略是否合理？
  - 全量覆盖
  - 增量合并
  - 条件合并
```

#### 4. 输出差异
```markdown
□ 最终输出是否保留了数据源标识？
  - 用户能否区分数据来自哪个源？

□ 输出格式是否统一？
  - 不同源的数据输出格式是否一致？
```

## 【新增】提示词逻辑验证四问

> 诊断提示词问题时的标准化提问框架

### Q1: 边界覆盖
**问题**：空值、最小值、最大值、异常值是否都有处理逻辑？

**检查方法**：
- 列出所有输入字段
- 为每个字段设计极端值测试用例
- 验证提示词是否能正确处理

### Q2: 条件完备
**问题**：if/else是否覆盖了所有分支？有无隐含的默认行为？

**检查方法**：
- 画出条件分支决策树
- 检查是否有遗漏的分支
- 验证else分支的处理逻辑

### Q3: 一致性
**问题**：示例输出与实际规则描述是否一致？

**检查方法**：
- 将示例输入代入规则
- 对比示例输出与规则推导结果
- 标记不一致处

### Q4: 可验证
**问题**：是否提供了明确的验证方法（如测试用例）？

**检查方法**：
- 是否能构造明确的测试用例？
- 预期输出是否可判定？
- 边界情况是否可验证？

## 附录：常见数据流故障模式

基于GitHub Issues统计的高发问题：

| 故障模式 | 症状 | 根因 | 诊断位置 | 解决方案 |
|---------|------|------|----------|----------|
| **静默失败** | 节点显示成功但无输出 | LLM超时/返回空 | L5模型层 | 检查超时设置，添加重试 |
| **类型不匹配** | Debug成功/API失败 | 前端未做输入校验 | L3参数层 | 统一输入格式校验 |
| **数据截断** | 日志显示不完整 | 并行深度限制/网络超时 | L6架构层 | 调整WORKFLOW_PARALLEL_DEPTH_LIMIT |
| **循环依赖** | 变量未按预期传递 | 变量绑定错误 | L2绑定层 | 检查变量引用关系 |
| **提示词溢出** | Token异常高 | 上下文未截断/循环引用 | L4提示词层 | 添加上下文长度限制 |
| **空值污染** | 计算结果异常（如负数） | 无数据与空数据混用 | L4提示词层 | 使用数据状态三层分类法 |
| **前缀冗余** | 输出无意义前缀 | 空值参与字符串拼接 | L4提示词层 | 条件化输出设计模式 |
