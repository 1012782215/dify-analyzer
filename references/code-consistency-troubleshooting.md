# 代码节点一致性诊断手册

> 当报错变量在当前代码中不存在时使用
> 覆盖：缓存未发布、上游节点、作用域、代码修改、运行时异常
> 配套 SKILL.md Phase 5.1.5 检查点

---

## 原因 A：代码已修改但未发布（最常见）

### 诊断步骤

1. 检查节点右上角状态
   - 显示"已发布" → 不是此原因
   - 显示"自动保存"或修改标记 → 可能是此原因

2. 点击"发布"按钮

3. 重新运行工作流

4. 观察错误是否消失

### 验证结果

- **发布后错误消失** → ✅ 根因：Draft 与 Publish 不一致
- **发布后错误仍在** → ❌ 不是缓存问题，排查其他原因

### 常见场景

- 用户在 Draft 中修改了代码，但忘记点击"发布"
- 自动保存成功，但手动发布未完成
- 多人协作时，其他人发布了旧版本

## 原因 B：上游节点未输出变量

### 诊断步骤

1. 找到引用该变量的上游节点
2. 点击该节点的"上次运行"
3. 检查输出中是否包含该变量

### 常见子原因

| 子原因 | 现象 | 解决 |
|--------|------|------|
| 上游节点执行失败 | 输出为空或报错 | 先修复上游节点 |
| 上游节点改了输出变量名 | 输出中有类似变量但名称不同 | 同步更新下游引用 |
| 上游节点输出类型变化 | 如 Array 变 String | 调整下游处理逻辑 |
| 上游节点未配置输出 | 节点配置中未声明该输出 | 在上游节点添加输出变量 |

### 快速检查清单

- [ ] 上游节点 ID 是否匹配
- [ ] 输出变量名是否完全一致（区分大小写）
- [ ] 上游节点是否在所有分支中都定义了该输出
- [ ] 变量名是否超过 30 字符（可能被截断）

## 原因 C：循环/并行结构变量作用域问题

### 循环节点场景

**问题**：变量在循环内定义，循环外引用时报"未定义"

```
循环节点
  ├─ 代码节点 A：const result = ...
  └─ 循环结束后
      └─ 代码节点 B：引用 result → ❌ 报错
```

**解决**：
- 使用循环节点的"输出变量"传递数据
- 或使用会话变量（Conversation Variable）跨循环传递

### 并行分支场景

**问题**：变量在分支 A 定义，分支 B 未定义，合并后引用时报错

```
并行节点
  ├─ 分支 A：定义变量 x
  └─ 分支 B：未定义变量 x
      └─ 合并后引用 x → ❌ 报错（分支 B 执行时 x 不存在）
```

**解决**：
- 确保所有分支都定义该变量
- 或使用默认值节点兜底

### If-Else 场景

**问题**：某分支未定义变量，但后续节点统一引用

```
If-Else
  ├─ If 分支：定义变量 x
  └─ Else 分支：未定义变量 x
      └─ 后续节点引用 x → ❌ 报错（Else 执行时 x 不存在）
```

**解决**：
- 所有分支都定义 x，或使用默认值
- 或使用变量赋值节点统一处理

## 原因 D：代码确实调整过（版本对比）

### 需要用户提供

1. **修改时间**（用于从 Dify 发布日志定位）
   - 例如："昨天中午"、"上周三"
   
2. **或修改前的代码片段**（用户直接粘贴）

### 分析流程

```
获取旧版本代码
    ↓
对比 diff（旧版本 vs 当前版本）
    ↓
判断：
  ├─ 旧代码有该变量 → 当前版本已修复
  └─ 旧代码也没有 → 与修改无关，排查其他原因
```

### 已修复时输出模板

```markdown
✅ 该问题在当前版本中已被修复

【改动点】
- 旧代码：第 X 行使用了 {{errorVariable}}
- 新代码：第 X 行改为 {{correctVariable}}

【修复原因】
变量名拼写错误 / 重构时遗漏 / 业务逻辑调整

【验证建议】
运行单节点测试确认报错不再出现
```

### 未修复时处理

→ 回到苏格拉底式排查，继续深挖其他原因

## 原因 E：Dify 运行时或缓存异常

### 可能原因

| 原因 | 说明 | 验证 |
|------|------|------|
| Sandbox 临时文件未更新 | Dify 执行的是旧缓存的 test.js | 清除缓存、重新保存发布 |
| 变量名在包装过程中被截断 | 长变量名被截断后匹配失败 | 检查变量名是否超过 30 字符 |
| Dify 版本 Bug | 如 Issue #22389：输出变量名含 "error" 触发误判 | 查阅当前版本已知 Bug 列表 |
| 节点配置损坏 | 配置 JSON 中存在非法字符或格式错误 | 导出 DSL 检查 |

### 验证步骤

1. 清除浏览器缓存，强制刷新（Ctrl+F5）
2. 重新保存并发布工作流
3. 如果仍报错，检查 Dify 版本是否有已知 Bug
4. 尝试将代码节点导出为独立脚本测试
5. 最后手段：删除节点重建

## 原因 F：自动排查流程（用户不确定时）

### 自动执行顺序

```
Step 1：检查发布状态
  ├─ Draft ≠ Publish → 提示发布（原因 A）
  └─ 已发布 → 继续

Step 2：检查上游节点输出
  ├─ 上游未输出 → 提示检查上游（原因 B）
  └─ 上游正常 → 继续

Step 3：检查作用域
  ├─ 变量在循环/分支内定义 → 提示作用域问题（原因 C）
  └─ 作用域正常 → 继续

Step 4：询问修改历史
  ├─ 用户确认近期修改 → 进入版本对比（原因 D）
  └─ 未修改 → 继续

Step 5：运行时问题
  └─ 提示缓存/运行时异常（原因 E）
```

### 终止条件

- 任一步骤定位问题 → 输出结论，结束排查
- 3 轮仍无果 → 建议人工介入，转入 Step 2 假设树

## 与现有机制的兼容矩阵

| 现有机制 | 兼容方式 |
|---------|---------|
| 苏格拉底式排查原则 | 每个分支都呈现假设，让用户验证 |
| 检查点（Iteration Gate）原则 | 明确标记为"检查点 1.5"，暂停等用户确认 |
| 渐进式输出（P1/P2/P3） | 默认只输出排查方向（P1），详细代码对比需用户确认（P2） |
| 兜底逻辑保护原则 | 不删除任何现有逻辑，只在旁边追加新分支 |
| 止损机制 | 最多 3 轮排查，无果则建议人工介入 |
