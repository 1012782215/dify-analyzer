# 代码节点故障模式库

> Dify 代码节点（JavaScript/Python）常见错误模式速查表。
> 用途：快速定位问题、选择修复策略。

---

## 错误模式速查表

| 错误模式 | 症状 | 根因 | 最小修复 | 保守优化 |
|---------|------|------|---------|---------|
| `ReferenceError: x is not defined` | catch 块崩溃，报变量未定义 | 变量名拼写错误（如 e1/e, err/error） | 修变量名，不动 catch 结构 | - |
| `SyntaxError: Unexpected token` | JSON.parse 失败，位置在某行某列 | 尾随逗号、分号代替逗号、中文全角标点 | - | 增加清理逻辑（去尾随逗号/替换分号） |
| `SyntaxError: Unexpected token '}'` | JSON.parse 失败在末尾 | 对象/数组末尾多了 `}` 或 `]` | 尝试去掉末尾一个 `}` 再 parse | 增加末尾校验 |
| `Bad control character in string` | JSON 解析报控制字符错误 | 字符串值内包含未转义的 `\n` 或 `\t` | 替换为 `\\n` / `\\t` | 统一转义控制字符 |
| Markdown 代码块污染 | 输出被 ` ```json ` ... ` ``` ` 包裹 | LLM 自动包裹代码块 | 剥离代码块标记 | 增加正则清理（贪婪匹配） |
| 单引号字符串 | `'key': 'value'` | LLM 输出 JavaScript 风格 JSON | 全局替换单引号为双引号 | 增加容错解析 |
| 中文全角标点 | `{"姓名"："张三"，"年龄"：18}` | LLM 输出中文标点 | 统一替换全角标点为英文 | 正则替换 `[：，；]` |
| JSON 截断 | `{ "a": 1`（有开头无结尾） | Token 不足，输出被截断 | - | 增加 max_tokens（建议 ≥ 8192） |
| 注释污染 | `{"a": 1, // 注释\n "b": 2}` | LLM 在 JSON 中加注释 | - | 去除 `//` 和 `/* */` 注释 |
| 分号代替逗号 | `"key": { ... };\n"next": ...` | LLM 混淆 JS 对象分隔符 | - | 精确正则：`};
"` → `},\n"` |
| 末尾多余 `}` | `...}}` 或 `...}]}` | LLM 多输出闭合括号 | 尝试 slice(0,-1) 后再 parse | 增加括号平衡校验 |
| 空值未引号 | `{"a": }` 或 `{"a": ,}` | LLM 输出不完整键值对 | - | 补全为 `"a": null` 或 `"a": ""` |
| Unicode 转义问题 | `\uXXXX` 序列解析失败 | 中文字符被截断或转义错误 | - | 确保 UTF-8 完整接收 |
| 科学计数法异常 | `{"price": 1.23e-4}` | 某些解析器不支持科学计数法 | 转换为字符串 `"1.23e-4"` | 使用宽松解析器 |

## 按症状快速定位

### 如果是 `JSON.parse` 失败

```
1. 检查是否有 Markdown 代码块 → 剥离 ```json ... ```
2. 检查是否有 think 标签 → 提取 </think> 后的内容
3. 检查是否有注释 → 去除 // 和 /* */
4. 检查是否有尾随逗号 → replace(/,\s*([}\]])/g, '$1')
5. 检查是否有分号 → 精确正则替换（见上表）
6. 检查括号是否平衡 → 统计 { } [ ] 数量
7. 检查是否有单引号 → replace(/'/g, '"')
8. 检查是否有中文全角标点 → replace(/[：，；]/g, c => ({'：':':','，':',','；':';'})[c])
9. 检查是否截断 → 末尾不是 } 或 ] → 增加 max_tokens
```

### 如果是 `ReferenceError`

```
1. 检查报错变量名（如 e1）
2. 检查同作用域内是否有同名变量（如 catch(e) 但用了 e1）
3. 只修变量名引用，不动 catch 结构
4. 检查是否有其他拼写错误（如 strs/str, json/jsn）
```

### 如果是执行结果不符合预期

```
1. 检查输入数据（Last run 记录）
2. 检查代码逻辑分支（是否进了预期分支）
3. 检查输出类型（字符串 vs 对象 vs 数组）
4. 检查是否有隐式类型转换（如 +'123' vs '123'）
```

## 修复策略选择指南

| 你的情况 | 推荐策略 |
|---------|---------|
| 急需修复，时间紧 | **最小修复** → 只修报错位置，保留所有原有逻辑 |
| 想彻底解决问题 | **保守优化** → 保留原有逻辑 + 增加清理/兜底 |
| 输入格式多变 | **替换逻辑** → 用正则精确替换 replaceAll，保留兜底 |
| 生产环境，不容出错 | **safeMode** → 只做清理不做正则修复，失败返回 error 不 throw |
| 经常出 JSON 问题 | **架构调整** → 增加 Fail Branch，接 LLM 自动修复 |

---

## 安全原则

- **严禁删除**：catch / else / endsWith / try-catch 嵌套 / 默认分支
- **保留注释**：标注"兜底/容错/降级"的注释是开发者意图，不可删
- **追加不替换**：新逻辑加在原有逻辑旁边，不覆盖
- **测试验证**：任何修改后必须单节点测试通过
